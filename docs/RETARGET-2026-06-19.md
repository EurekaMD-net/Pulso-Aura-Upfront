# Deployment re-target: base crm-azteca → Pulso-Aura-Upfront (2026-06-19)

## Why

The deployable product is **Pulso-Aura-Upfront** (Aura corpus + Modo Cierre closing
companion), but the live `agentic-crm` deployment — and Pulso-Aura's **own** tracked
deploy scripts — were hardcoded to the **base `/root/claude/crm-azteca`**. So even
"deploying Pulso-Aura" would have run the base engine against the near-empty base DB.
This session re-pointed the whole stack at Pulso-Aura.

## ⛔ Operator gates before it runs (cannot be done from the agent side)

1. **Complete Pulso-Aura's `.env` provider config.** Pulso-Aura's `.env` was never
   populated for deployment ("service UNDEPLOYED") — it has `INFERENCE_PRIMARY_KEY`,
   `EMBEDDING_MODEL/URL`, raw `GROQ/MINIMAX/OPENAI` keys, but is **missing
   `INFERENCE_PRIMARY_URL`, `INFERENCE_PRIMARY_MODEL`, the `INFERENCE_FALLBACK_*` trio,
   the `INFERENCE_*` tuning vars, and `HINDSIGHT_LLM_*`.** `crm/src/bootstrap.ts:99
validateEnv` **hard-aborts** the boot without `INFERENCE_PRIMARY_URL` + `_MODEL`.
   The base `crm-azteca/.env` has a proven-working set (Fireworks `minimax-m2p7`, 200/599 ms).
   Recommended one-liner (run via the `!` prefix; systemd EnvironmentFile takes the LAST
   value, so the appended block wins over the partial existing `INFERENCE_PRIMARY_KEY`):
   ```
   grep -E '^(INFERENCE_|HINDSIGHT_LLM_)' /root/claude/crm-azteca/.env >> /root/claude/Pulso-Aura-Upfront/.env
   ```
   Then **restart hindsight** so its LLM keys bake in: `crm-ctl hindsight-stop && crm-ctl hindsight-start`.
2. **WhatsApp pairing** (`store/auth` absent): `crm-ctl reauth` (pairing-code in
   `store/auth-status.txt`). Pulso-Aura's number is the operator's call — **must NOT be
   the salones/Gilda PROD number `525640501088`** (one number ↔ one bot).
3. **WhatsApp inbound is still unresolved on `522205847540`** (sends, can't receive — the
   prior session's account-side blocker). Deferred per request; revisit after the above.
